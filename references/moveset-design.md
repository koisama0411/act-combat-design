# 招式集设计法 · Moveset Design

> **本方法论不预设"必须有哪几招"。** 一个角色要覆盖哪些招式类型，由你项目的约定决定 → 先填 [conventions/TEMPLATE.md §2 招式集](../conventions/TEMPLATE.md)。
> 本文给的是：① ACT 常见招式类型的**设计空间**；② 每类招式的**设计手艺**（怎么做才好用、好看、有记忆点）。挑你项目需要的类型，套用对应手艺。

---

## 一、先决定你的"招式集契约"

每个项目都该有一份"每个角色至少要有哪些招"的清单——这是美术 / 程序估工作量的基准，也是玩家的操作预期。

**怎么定**：
1. 看你的**核心战斗循环**需要哪些动词（攻击 / 闪 / 格 / 蓄力 / 切人 / 处决…）。
2. 从下面的"招式类型菜单"挑出**必备项**，其余列为"可选 / 角色特色"。
3. 定**衔接 / 取消规则**（连段能否闪避取消？蓄力能否被打断？大招有没有霸体？）。

> 轻量手游可能只需 4–5 类；硬核动作游戏可能十几类。**没有标准答案，只有适不适合你的项目。**

---

## 二、招式类型菜单 + 各类设计手艺

下面每类给"设计要点"。**示例命名用 PascalCase（`Attack_01` 等），你项目按自己的命名规范替换**（见 [conventions/TEMPLATE.md §6](../conventions/TEMPLATE.md)）。

### 1. 普攻连段（almost always 必备）

- **段数**：短小精悍 3 段 / 主流 4 段 / 长 5 段。段数越多越要防"中段无聊"。
- **节奏**：起手 1–2 段轻快流畅，中段渐重，**末段最重**（常出击退 / 击飞），给连段一个"标点符号"。
- **位移**：每段带少量位移（地面推进 / 跃空 / 后撤），让连段有空间感，也帮玩家贴身。
- **判定范围**：轻段窄、重段宽。
- **设计取舍**：同一段是否要"上下半身分离"（为瞄准 / 多方向）vs 一体动作——见下方"普攻方案讨论"。

### 2. 重攻击 / 蓄力攻击

- 长按蓄力的，拆 **pre（前摇）+ charge（蓄力 loop）+ release（释放）+ recover（后摇）** 四段。
- **蓄力越久效果越强**要有清晰的"档位反馈"（音效升调 / 蓄力光效分级 / 角色姿态变化）。
- **蓄力期不能是纯等待**：允许移动、允许被完美闪避取消并保留蓄力进度，否则违反"蓄力期无操作"反模式（见 [anti-patterns.md](anti-patterns.md)）。

### 3. 核心特殊技（角色机制载体）

特殊技通常是**角色差异化的核心**，跟设计原型直接挂钩。常见形态：
1. **长按蓄力体**：pre + charge loop + 释放 + recover；蓄力越久越强。
2. **资源条触发替换**：满资源时特殊技变"强化版"。
3. **变形态触发**：释放后改变普攻形态 / 进入新状态。
4. **短长按二分**：短按单发 / 长按全发，效果不同。
5. **持续 loop**：释放后进入循环，跟随摇杆缓移。

> 怎么挑机制载体 → 见 [character-archetypes.md](character-archetypes.md)（原型库）。

### 4. 闪避 + 闪避后续

- **闪避**：基础位移规避；很多项目给"完美闪避窗"（极限闪避触发子弹时间 / 反击窗）。
- **闪避攻击**：普通闪避后接的攻击，可复用普攻 1 段或单独动作。
- **闪避反击**：完美闪避后接的进阶攻击，常带短距推进 + 更强控制。
- "闪避接特殊技 / 闪避取消蓄力保留资源"是给高手的隐藏操作空间，能显著抬高操作上限。

### 5. 防御 / 格挡 / 弹反 / 拼刀

- 若项目有此类，先对齐**视觉预警契约**（什么预警可格 / 可弹）→ 见 [conventions/TEMPLATE.md §4](../conventions/TEMPLATE.md)。
- 弹反对撞常拆 **start（判定窗，给反应时间）+ success（对撞 loop）+ attack（反击，常多 hit）+ end（后摇）**。
- **地面版 / 空中版是否都要**：取决于你项目有没有空战。有空战就两套都配，没有就只地面——**别无脑照搬"必须地空各一套"**，那是某些项目的选择，不是通用铁律。

### 6. 空中攻击

- 下劈 / 下砸 / 水平扫射等；落地常带 AOE。
- 可单段，也可多段（如一段水平、二段斜下）。

### 7. 终结技 / 大招

- 单段高爆发 vs 多段表演（3–4s 演出）。
- 常配合"召唤 / 砸落 / 全屏冲击"等高强度演出 + 立绘 / 运镜。
- 前摇给警示、打击给爽感、后摇别太长（"大招太长不可打断"是反模式）。
- **跟"团队联动技"区分**：大招通常是自身能量满；联动技是团队 / 后台能量满。

### 8. 团队联动技 / 切人技（仅团队制需要）

- 多角色协同时的支援 / 接管招。形态由你项目的配队模型决定（切人联动 / 后台挂状态 / 共享槽消耗）。
- 单角色游戏**直接跳过这类**。
- 设计思路 → 见 [team-comp-principles.md](team-comp-principles.md)。

## 三、命名规范（示例，定义你自己的）

一套统一命名让美术 / 程序 / 策划对齐。示例（PascalCase 风格）：
- 普攻：`Attack_01` … `Attack_0N`
- 蓄力技：`Skill_Charge_Pre` / `_Loop` / `_Release` / `_Recover`
- 闪避：`Dodge` / `Dodge_Attack` / `Dodge_Counter`
- 弹反：`Parry_Start` / `_Success` / `_Attack` / `_End`
- 空攻：`JumpAttack_01`
- 大招：`Ultimate_01`
- 子段后缀：`_pre` / `_charge` / `_loop` / `_release` / `_recover` / `_air` / `_start` / `_end`

> 这只是示例风格。你项目用 snake_case、用中文功能名、用别的层级都行——在 [conventions/TEMPLATE.md §6](../conventions/TEMPLATE.md) 定下来即可。

## 四、"普攻方案讨论"——决策留痕

普攻这种核心招常有取舍（如"表演性优先" vs "瞄准 / 功能优先"，"上下半身分离" vs "一体动作"）。做这类决策时，**建一个方案对比表留档**：

```
问题：<要决策的问题>
---
方案一：<名字> / <描述> / 优点 / 缺点
方案二：<名字> / <描述> / 优点 / 缺点
---
结论：<选哪个，为什么>
```

废弃方案不要删，挪到"垃圾箱"留痕（见 [design-doc-structure.md](design-doc-structure.md)）——方便日后追溯"为啥不那么做"。

---

## 五、设计前必做：找最近的原型当 baseline

设计一个新角色的招式集时，**先比对你项目已有角色 / 业界原型里最像哪一个**（按定位 / 武器 / 核心机制），把它的招式结构当 baseline，再做差异化改造。**别从零拍脑袋。**
