# 文档结构模板

二次元 ACT 战斗设计文档(xlsx / 多 sheet)的标准 sheet 集和字段定义。设计新角色 / Boss 时,以此为骨架。各字段是行业通用的"该想清楚的事",不绑定任何具体引擎或项目格式 —— 用 xlsx、Notion、Markdown 表格承载都可以。

---

## 玩家角色 xlsx

### Sheet 1: 战斗玩法 (必填)

| 模块 | 字段 / 内容 | 备注 |
|---|---|---|
| **基本信息** | 角色名称 / 分类轴(见备注) / 使用武器 / 定位 / 上手难度(★) | 分类轴由你的项目自定义,示例:职业 × 定位 × 稀有度,详见 [角色原型库](character-archetypes.md) 与 [项目约定模板](../conventions/TEMPLATE.md) |
| **设计思路** | 大段描述(蓝本 + 核心机制 + 风格定调) | 100-300 字,讲清楚"为什么这么设计" |
| **核心记忆点** | 1.X / 2.X | 1-3 条,玩家容易记住的招牌设计点 |
| **主题风格** | 一句话 | 视觉/感觉定调,如"硬朗刚劲的会武术的保镖" |
| **类似玩法** | 1-2 个游戏角色参考 | 如"火影忍者-咏春宁次"、"DNF 神枪/枪炮" |
| **推荐配队** | 1-3 个组合 | 给队伍上下文(配队设计见 [配队协同原则](team-comp-principles.md)) |
| **被动机制** | 描述 + 触发条件 | 入场触发 / 数值条件触发 |
| **养成 / 进阶提升** | 1.X / 2.X / 3.X | 进阶方向(自身/全队/机制扩展);具体养成形式(命座 / 技能树 / 天赋等)由项目自定义 |
| **主要玩法 / 循环手法** | 短轴 + 长轴描述 | 玩家实际操作的 ABC 循环 |
| **技能设计** | 技能类型 / 玩法描述 / 技能定位 / 玩法示意 | 按你项目定义的招式集逐个填(招式集设计见 [招式集](moveset-design.md)) |
| **垃圾箱** | 废弃方案存档 | 设计取舍留痕,**强烈建议保留** |

### Sheet 2: 动作需求 (必填)

每行一个动作资产:

| 字段 | 说明 |
|---|---|
| 动作命名 | 中文功能名,如"普通攻击"、"特殊技"、"闪避反击" |
| 资产命名 | 英文资产名,见 [招式集](moveset-design.md) 命名规则 |
| 动作说明 | 动作功能性说明 |
| 循环 | 是 / 否(loop 标识,如蓄力/移动 idle 用是) |
| 位移 | 距离 / 速度(如"3 格"、"1 秒 5 unit") |
| 判定范围 | 打击判定范围(如"3X3"、"6X3"、"4X3"、"8x5") |
| 对齐说明 | 首帧衔接 Pose(如"idle"、连招衔接帧、蓄力起手帧) |
| 前摇(帧) | 第一次打击前的时间 |
| 打击持续(帧) | 单段攻击通常 3-6 帧 |
| 后摇(帧) | 自然恢复时长 |
| 玩法信息 | 跟 Sheet1 技能设计的对应描述 |
| 示意说明 | 视频/图文链接 |
| 动作草图 | 参考图链接 |

**玩家角色帧数指导(示例帧数预算,30fps 语境;按你项目帧率替换)**:
- 普攻轻招: 前摇 4-8f / 打击 3-6f / 后摇 4-6f
- 普攻重招(最后段): 前摇 6-10f / 打击 5-8f / 后摇 7-10f
- 特殊技: 前摇 6-10f / 打击 10-15f / 后摇 5-7f
- 蓄力体: pre 5f / charge loop 5f / 释放 10-13f / recover 5f
- 弹反对撞: start 18-24f(给玩家反应窗) / success loop / attack 3-6f 前摇 + 30-40f 持续 + 6-10f 后摇
- 闪避攻击/反击: 前摇 3-5f / 打击 7-10f / 后摇 4-9f
- 空中攻击: 前摇 3-10f / 打击 5-10f / 后摇 5-8f
- 大招: 前摇 7-40f / 打击 20-60f / 后摇 10-15f

### Sheet 3: 功能需求 (按需)

格式:
```
功能 N
用途: ...
期望: ...
方案:
  1. 新增通知 / 新增状态 / 新增逻辑(名称)
  2. 参数列表
```

**示例案例**(借这两类需求最快理解这个 sheet 怎么写):
- 一个近战拉拽型角色:CustomMove 弧线轨迹通知 / PullTarget 推动目标通知 / 区域检测确定技
- 一个远程长按持续型角色:`<移动并攻击>` 状态 + 长按方向跟随 + 基于目标角度改变技能方向 + 持续射击时长通知

### Sheet 4: HUD 需求 (按需)

资源条 / 形态切换 UI / 命中标记图标等。用图示意每种状态(idle / 资源积累中 / 满 / 释放后)。

**示例案例**:双形态机甲角色的形态能量条 / 弹药量计数 HUD —— 每个数值都要有对应的视觉状态。

### Sheet 5: 普攻方案讨论 (按需,但强烈推荐)

多方案对照表,做设计取舍时**强烈建议留这个 sheet**,方便追溯。

格式:
```
问题: <要决策的问题>
---
方案一: <名字>
<描述>
优点: ...
缺点: ...
---
方案二: <名字>
<描述>
优点: ...
缺点: ...
---
结论: <选哪个,以及为什么>
```

**示例案例**:远程角色普攻方案讨论(方案一全向瞄准 vs 方案二表演性优先,结论:先试方案一)。

## 角色 · 审阅级 HTML 完整稿范式

> 上面的「玩家角色 xlsx」是**结构化字段定义**(该想清楚的事);本段是进阶的**审阅级 HTML 产出范式** —— 把那些字段组织成一份可评审 / 可协作 / 可导出回传的 HTML 文档。两者并存:先按 xlsx 字段想清楚内容,再按本范式落成 HTML。

### 两种规格:分稿 vs 完整稿

一个角色的设计产出有两种规格,**按需求选**:

| 规格 | 形态 | 适合 | 样板 |
|---|---|---|---|
| **① 分稿** | **战斗设计稿** + **效果与养成稿** 两份**分开**,各自精简 | 「先出设计稿、之后单独再出养成稿」的分步节奏 | [`output/characters/cang-lan/`](../output/characters/cang-lan/)(苍岚:`苍岚-战斗设计.html` 战斗设计 + `苍岚-养成与效果.html` 效果养成,两份分开) |
| **② 完整稿** | **一份完整**、9 章,含数值建模 / 配队 / 资源清单等全部内容 | 「要一份完整的」的一站式节奏 | [`output/characters/jin-yu/`](../output/characters/jin-yu/)(烬羽:`烬羽-战斗设计及养成效果.html` 一份到底 9 章全 + 数值建模 5 图 + `md/烬羽-战斗设计及养成效果.md` 源稿;骨架取自模板 [`templates/blank-character-design-doc.html`](../templates/blank-character-design-doc.html)) |

**怎么选**:
- 用户**明示**(说「分步出 / 先设计稿」或「要完整的一份」)→ 按明示走。
- 用户**没明示** → **主动询问一句**:「要分步出(先战斗设计稿,之后单独出养成稿)还是一份完整稿?」拿到答复再动手,别替用户拍板。

### 设计稿文件命名规范

角色设计稿的文件名(html + 同名 md)按**稿件内容**统一命名,前缀一律是**角色名**,后缀标明稿件类型。落地生产清单(资源需求)用独立后缀,不算设计稿:

| 稿件类型 | 内容 | 文件命名 |
|---|---|---|
| **只战斗设计**(分稿之一) | 只讲战斗玩法 / 招式 / 手感,不含养成效果 | `角色名-战斗设计`(`.html` + `md/角色名-战斗设计.md`) |
| **只养成与效果**(分稿之一) | 只讲技能附带效果 / 命座 / 养成物 / 数值投放 | `角色名-养成与效果`(`.html` + `md/角色名-养成与效果.md`) |
| **完整稿**(战斗 + 养成效果) | 一份到底 9 章全,战斗设计 + 养成效果合并 | `角色名-战斗设计及养成效果`(`.html` + `md/角色名-战斗设计及养成效果.md`) |
| **资源需求**(落地生产,非设计稿) | 逐资产逐帧的生产级明细(动作 / 特效 / 程序) | `角色名-资源需求`(`.md` + `.xlsx`) |

- **分稿** = `战斗设计` + `养成与效果` 两份分开;**完整稿** = `战斗设计及养成效果` 一份合并;**落地** = `资源需求`。
- 范例:**苍岚**走分稿,故有 `苍岚-战斗设计.html` + `苍岚-养成与效果.html` 两份;**烬羽**走完整稿,故为单份 `烬羽-战斗设计及养成效果.html`(9 章全)。两者的 md 源稿同名放各自 `md/` 子目录。
- 🔴 命名只反映**稿件内容**,不要用「审阅稿」「演示」等过程词当文件名 —— 「审阅稿」是稿件的**协作形态**(评审 / 留痕 / 导出),不是稿件类型,所有设计稿都自带这套交互层,不进文件名。

### 完整稿 9 章范式(对应模板 `templates/blank-character-design-doc.html`)

| # | 章节 | 通用性 | 内容 |
|---|---|---|---|
| 壹 | 设计定位与哲学 | 🟢 通用必有 | 基本信息表(名 / 分类轴**按项目** / 武器 / 定位 / 上手)+ 设计哲学 + 核心记忆点 + 主题 + 类似玩法 |
| 贰 | 核心机制 | 🟢 通用必有 | 这角色的核心玩法机制(具体机制按角色填)+ 核心循环短轴 / 长轴 |
| 叁 | 招式详解 | 🟢 通用必有 | 按你项目**招式集**(conventions 定义)逐招 |
| 肆 | 资源系统(深入) | 🟡 **按角色** | 有特殊资源 / 状态 / 形态系统的角色拎一章深入;**没有则整章删除** |
| 伍 | 养成 / 命座 | 🟢 通用必有 | 养成形式**按项目**(命座 / 技能树 / 天赋…) |
| 陆 | 配队协力 | 🟡 **按项目** | 三人队 / 有协同机制的项目才有;**单机 / 买断制项目整章删除** |
| 柒 | 数值建模 | 🟢 通用必有 · 🔴 硬标准 | 5 张图 + 三段注解 + 平衡自检表(详见下「🔴 数值建模硬标准」) |
| 捌 | 资源清单 | 🟢 通用必有 | 6 类协作表(动作 / 特效 / 音效 / UI·HUD / 场景·演出 / Timeline·Prefab)+ 汇总卡片 |
| 玖 | 设计自检 | 🟢 通用必有 | 5 维质量自评(见 [`design-aesthetics.md`](design-aesthetics.md) § 一)+ 元规则命中(见 [`meta-rules.md`](meta-rules.md))+ 垃圾箱(废弃方案留档) |

> 🟡 的两章(肆 / 陆)是「按角色 / 按项目」可选项 —— 用不到就整章删除,并同步模板里 `SECTIONS` 数组。

> 📊 **完整稿的图表配置 = 操作 / 资源 / 数值三类可视化都要,不只数值章配图**:完整稿至少配 **贰 · 操作循环示意图 + 肆 · 资源流转示意图 + 柒 · 数值建模 5 图**(共 2 + 5 = 7 张,肆章若整章删除则该图随之去掉)。① **贰 · 操作循环示意图**:核心招怎么循环(普攻 / 特殊技 / 引爆 / 资源)+ 穿插招 / 节点怎么接入,三区结构(核心循环 / 穿插招 / 节点配队);② **肆 · 资源流转示意图**:各资源一个储库,上=进(来源)、下=出(消耗),互锁关系用箭头连;③ **柒 · 数值建模 5 图**:见下「🔴 数值建模硬标准」。三类图都 `role="img"` + `aria-label`,自动接到图表点击放大 lightbox。模板 `templates/blank-character-design-doc.html` 的贰 / 肆章已带空图骨架,样板见 `output/characters/jin-yu/烬羽-战斗设计及养成效果.html`(贰图 1 操作循环 + 肆图 2 资源流转 + 柒 5 图)。

### 🔴 数值建模硬标准(防缩水,单独强调)

**无论选完整稿、还是选分稿里的「效果与养成稿」,数值建模都是必有项,不许缩水。** 必须做到:

1. **配齐 5 张图**(缺一不可):
   - ① 命座 / 养成曲线(累计强度指数随养成层级)
   - ② 逐级边际(每级边际增益 %,看质变跳点是否合理)
   - ③ 增伤区稀释(加法 bucket 自堆增伤的边际递减 `S ÷ (1+面板)`,说明角色走哪条赛道)
   - ④ 场景伤害梯度(单体 / 群体 / 破韧 / 持续等场景相对强度,体现定位)
   - ⑤ 机制专属曲线(角色核心机制的收益 vs 代价随档位变化)
2. **每图配「怎么算 · 为什么 · 影响」三段注解**(模板里是 `.anno-box`)。
3. **末尾配平衡自检表**(满级养成总梯度 / 增伤边际 / 场景梯度 / PVP 比 / 机制收益代价 等,对照健康阈值给 ✅ / ⚠️ / ❌ 结论)。
4. **只写数值表、不配图 = 不合格。**

建模口径:乘区式(技能 × 增伤 × 暴击,守方乘区归一化为 1),数值为相对指数 / 倍率(已归一化),绝对伤害需代入项目真实面板复算;PVP 类临界点务必用**项目真实公式**复核,别用简化式拍板。健康阈值**按项目 / 定位 / 品类(买断 vs gacha)取用**。方法论详见 [`numbers-and-progression.md`](numbers-and-progression.md);分稿里的养成 / 效果稿写法详见 [`effect-progression-design.md`](effect-progression-design.md)。

### Sheet 2: 动作需求 (必填)

**风格定调段(放 sheet 顶部,示例数值为 30fps 语境,按你项目帧率替换)**:
> ① 前摇要让玩家"看到 → 反应",简单招 18-22f / 警示招 25-30f / 大招 35f+
> ② 命中帧 1-3f 锐利单段
> ③ 后摇按重量分级,轻招 10-15f / 重招 20-30f / 大招 25-35f
> 复用项尽量沿用现有特效资产,仅做颜色/强度调整

字段(每行一招):

| 字段 | 说明 |
|---|---|
| 动作用途 | 中文功能名 |
| 资产名称 | 英文资产名(示例命名规范如 `skill_01` ~ `skill_11`,定义你自己的) |
| 类型 | 标准刀 / 蓄力重击 / 位移 / 表演 / 后撤 / 范围攻击 / 原地 buff 等 |
| 复用方式 | 复用角色资源制作 / 新增 |
| 动作说明 | 拼接段说明(如"角色普攻第 3 段 + 第 4 段") |
| 循环 / 位移 / 判定范围 / 对齐 Pose | 同玩家角色 |
| 前摇 / 打击持续 / 后摇 | (帧) |
| 备注 / 参考图 | 调整理由 |

**通用动作集(建议所有 Boss 都覆盖到的态,示例命名)**:
```

hitground_light_up / hitground_light_down / hitground_heavy
hitdeath_ground / hitdeath_strike
hitfloat_up / hitfloat_airloop / hitfloat_fallloop / hitfloat_onground / hitfloat_lieloop / hitfloat_getup
```

### Sheet 4: 程序需求 (按需)

字段:**需求简述 / 实现方式 / 优先级**

**示例案例**(一个"以增益机制为核心"的 Boss):
- 技能加速机制 — buff 实现,buff 可配置播放倍率
- 次数盾 — buff 层数实现,每收到一次攻击减少一次层数,buff 持续期间免疫所有伤害

## 起手策略

设计新角色 / 新 Boss 时:

1. **先 Read 一份最接近的参考样本**(基于设计输入选最像的现有角色/Boss),用作骨架
2. **按本文档 sheet 集和字段一一对应**,先用 markdown 草稿展开
3. **垃圾箱 sheet 留废弃方案存档**,设计过程中的取舍不要删,挪过去
4. **决策有取舍**(如普攻是表演性优先还是瞄准优先)时,建一个独立"XX 方案讨论 sheet",留方案对比表
