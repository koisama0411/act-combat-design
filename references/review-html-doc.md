# 审阅级 HTML 设计文档范式

输出 Boss / 角色战斗设计稿时,用这套「审阅级 HTML 文档」范式 —— 单文件 HTML,自带多人协作交互,可在浏览器内直接评审、填产出、回传。

**为什么用 HTML 而非 markdown / docx**:
- 浏览器双击即开,零依赖
- 可交互:审稿划除资源 / 行内编辑 / 写评审备注 / 贴产出截图视频 / 跳章节
- 单文件,所有数据(含图/视频 base64)嵌在一个 .html 里,发送备份方便
- 支持「导出 → 回传 → 加载」的多人协作闭环

**空白模板**:本仓提供一份空白模板 [`templates/blank-design-doc.html`](../templates/blank-design-doc.html) —— 它是**从一份实战打磨过的审阅稿直接提取**的完整版:头部固定模块(封面 / 协作说明 / 改动历史 / 目录)+ 章节空容器(默认含**审阅向 `#overview` / 落地向 `#detail` 分章**)+ 冷蓝 CSS 基底 + **完整富媒体协作层**(导出修改版 / sessionStorage + IndexedDB 大文件 / 粘贴拖拽图片视频 / B站YT iframe / 资源表增删改 / 改动历史 / 浮动两级目录)。新设计稿**复制它,清空章节内容、改标题和 `LS_PREFIX` 即可**;交互层(全部 CSS/JS)整套保留不动。

> 模板**已是完整版**——富媒体协作层(IndexedDB 大文件 / B站YT iframe / 改动历史富交互 等)都已内置(从实战稿提取)。[审阅级 HTML 组件实现](review-html-components.md) 解释这些组件的实现与陷阱,**改模板 / 复用组件 / 排查 bug 时**才查,出新稿不必从零搭。

📎 **本文是范式 + 用法**。组件实现细节(呈现层组件 / 审阅交互组件 / 协作富媒体 / 存储架构 / 四陷阱 / 评审反馈标记套件)拆到 [审阅级 HTML 组件实现](review-html-components.md),**只在重写交互层 / 修模板 bug / 复用组件时查**;出新设计稿不用看它(交互层整套从模板复制不动)。

## 〇、生成新文档的标准流程

1. **Read 空白模板**通读一遍(结构 + CSS + JS),或读一份你之前做好的成稿当参考
2. **复制**成新文件,按对象类型归档到你的产出目录下的子文件夹:
   - 角色稿 → `<你的产出目录>/角色设计/<角色名>_设计.html`
   - 怪物 / Boss 稿 → `<你的产出目录>/怪物设计/<名字>_Boss设计.html`
3. **改 3 处标识**:
   - `<title>` 和 cover 区
   - JS 顶部 `const LS_PREFIX = '<doc-prefix>-';` → 换成本文档专属前缀(如 `'cl-boss-'`),避免跟其他文档 sessionStorage 冲突
   - 导出文件名模板里的对象名
4. **清空七章节正文**,按结构重新填
5. **交互层整套保留不动**——操作列 / 协作列 / 参考列 / 资产名称列 / **改动历史区** / **左侧坞(章节导航+协作工具+置顶)** / **两级目录(浮动+顶部)** / **复用资源沙青降权** / 评审备注 / 媒体弹窗 / 导出 / 大文件存储
6. **单场 / 双场处理**(模板默认单形态单场,`肆` 自带一个**可删的 Tab**):
   - **先判断是不是真双场战**:双场战 = **同一对象、机制差异化的两场**(如「骗术场 vs 觉悟场」,招式池/调性不同)。若是**同 Boss 多次出现但机制完全一样**(两场招式池/反制相同,只是剧情位置不同),那仍是**单场** —— 放进 `贰` 段用表格说明剧情位置即可,不要套双场 Tab
   - **单形态单场**:删掉 `肆` 里的 Tab 切换(只留单 panel);**Tab 删了后两级目录的 P1/P2 分组逻辑会自动失活**(扫不到 `.tab-panel` 自然不分组),无需手动改分组 JS
   - **双场战 / 多形态**:保留 `肆` 的 Tab,两个 panel 各放一场 / 一形态,标签改成对应名称;需并排对比时在 `肆` 顶部加一张对比表
   - ⚠️ **删 section 后必须三处同步**(否则目录跟内容对不上):① 章节 `<h2>` 编号**前推**(删了叁,后面 肆→叁 / 伍→肆 … 不要跳号) ② JS 里 **hard-coded 的 `SECTIONS` 数组**(目录是从这个数组生成的,不是纯动态扫 h2,详见 [组件实现 § 二](review-html-components.md)) ③ 各段 `📝 评审备注 · X` 的汉字编号。**加/拆 section 同理**(如详细设计拆两章):h2 号顺延、`SECTIONS` **加**条目、评审备注号顺延;新 `note-aside` 用**唯一 `data-key`**(+配对 `data-clear`),`<!-- N. -->` 注释号对齐
   - ⚠️ **复制模板后清空 `<script id="__embed_state__">`** —— 模板/旧稿里可能残留上一个对象的协作数据(deleted-rows / added-rows / note 等),不清会被当成本文档数据加载

## 一、七章节标准结构(可裁剪)

```
壹 · 设计哲学            #philosophy   — 一段话讲清「角色为什么是这样」 + 角色弧光
贰 · 剧本时序总览        #plot         — timeline 节点串剧本 ID + 对话框(有剧情才需要)
叁 · 核心机制 + 招式一览(审阅向)  #overview   — 机制概览卡 + 招式卡(看这章就懂怎么打)
肆 · 落地配置细节(落地向)        #detail     — 行为/数值/几何/收益曲线/时间线/对白;多形态·多阶段·双场在此用顶部 Tab 切换,单形态删 Tab
伍 · 叙事-机制双绑定     #bind         — 「叙事+机制双绑定」元规则落地表
陆 · 资源清单            #assets       — 6 类资源 × 复用/新增 矩阵(协作核心区)
柒 · 元规则命中          #metarules    — 命中元规则清单 + 场景
```

**章节编号用汉字**(壹/贰/叁/肆/伍/陆/柒)—— 提升正式感,且跟内容里 skill1/skill2 的阿拉伯数字区分。

> **这 7 章就是空白模板 `blank-design-doc.html` 的默认章节**(`叁 #overview` 审阅向 + `肆 #detail` 落地向是默认拆分)。**双场战变体**:把 `肆` 的 Tab 用于"中段战 / 末段战"两场(见 § 三 双场 Tab 调色),需并排对比时在 `肆` 顶部加一张双场对比表;单形态单场则删掉 `肆` 的 Tab。

每个 `<section>` 末尾挂一个 `<aside class="note-aside">` 评审备注区。

**推荐默认结构:审阅向 / 落地向 两章(模板已内置)**

把"看设计"和"照着落地"拆成两章,是本范式**推荐的默认**(空白模板已内置 `#overview` + `#detail`)。**审阅人要的**「核心机制 + 技能描述」放审阅向;**落地配置(人 / AI)才用的**「行为 / 参数 / 收益曲线」放落地向:

```
叁 · 核心机制 + 招式一览(审阅向)  #overview  — 机制概览卡 + 招式卡
肆 · 落地配置细节(落地向)        #detail    — 行为/数值/几何/收益曲线/时间线/对白
```

- 两章开头各加一句互指导航语;审阅向引数值用「详见『肆』」指过去,不堆数字。
- 只搬运/提炼,不新增设计;精确数值留落地向。
- 模板已默认拆好两章;内容简单到不值得分时再合并。改章节记得同步 `SECTIONS`(见 § 〇.6)。

### 一(b) 头部固定模块(在 壹·设计哲学 之前)

按 DOM 顺序四块,所有审阅级稿件共用,不属于「七章节」也不属于交互层,**给读者第一屏就看到「文档约定 + 改动轨迹 + 导航入口」**:

1. **`.cover` 封面**:tag + h1 + sub + meta 元数据网格(原作蓝本 / 人设标签 / 武器形态 / 关卡 / 叙事钩 / 资源策略)
2. **`.collab-help` 协作流程说明**:本文档协作机制 + 两个协作列谁填 + 富媒体支持范围
3. **`.history-block` 改动历史**(6 列表格,详见 [组件实现 § 二「改动历史区」](review-html-components.md))
4. **`.toc` 内联目录**(两级树,chevron 可展开扫所有 h3 子项,详见 [组件实现 § 二「左侧坞 + 两级章节导航」](review-html-components.md))

## 二·b、文案标点规范（中文全角,**铁律**）

**所有文案标点统一用中文全角**——英文半角标点（`,` / `.` / `:` / `;` / `!` / `?` / `(` `)` / `"`）在中文段落里显得拥挤、夹生,破坏古风排版的节奏感。

| 英文半角 | 改用中文全角 | 备注 |
|---|---|---|
| `,` | `，` | 句中停顿 |
| `.` | `。` | 句末 |
| `:` | `：` | 解释/列举前 |
| `;` | `；` | 并列分句 |
| `!` | `！` | 感叹 |
| `?` | `？` | 疑问 |
| `( )` | `（ ）` | 括注；**纯英文/代码 括注里保持半角** |
| `"..."` | `"..."` 或 `「...」` | 引语；台词用 `「」`、强调用 `""` |
| `...` | `……` | 省略号双对 |
| `-` 句中破折 | `——` | em-dash 双连 |
| `-` 分隔列举 | `·` 或 `、` | 中间圆点 / 顿号 |
| `/` 分隔 | `/` 保留 | 列举/或关系常见,不强求换 |

**例外（保持半角）**:
- 文件名 / 路径 / URL:`templates/blank-design-doc.html` / `https://...`
- 代码 / 配置 / 函数名 / 类名:`init()` / `.tab-panel.p1` / `LS_PREFIX`
- 数值+单位 / 比例 / 公式:`30px` / `HP 50%` / `8s` / `1.3fr`
- 英文专有名 / 缩写:`Boss` / `Spine` / `BGM` / `VFX` / `SFX`
- 表格 markdown 分隔符 / 列表符号 / code-fence

**实操要点**:
1. **新写文档**:直接用中文标点写,不要「先写英文再回头改」
2. **改模板拷贝时**:用 IDE 全词替换需谨慎——代码段会误伤；推荐用 regex `(?<=[一-龥]),(?=[一-龥\s])` 这类锚定中文字符附近的半角逗号,逐个 confirm
3. **行尾收束词后**:习惯句号 `。`,**不留半角句号或不留终止符**
4. **括号嵌套**:外中内英可以——`（详见 config.json）`

## 三、双场 Tab 调色机制(双场战专用)

第肆节用 `.tab-btn` + `.tab-panel.p1/.p2` 切换,每场一套独立调色。

```css
.tab-panel.p1 h2 { color: var(--p1-accent); }
.tab-panel.p1 h3 { color: var(--p1-accent); }
.tab-panel.p1 .philosophy { background: linear-gradient(to right,var(--p1-bg),var(--paper)); }
.tab-panel.p1 th { background: var(--p1-card); }
.tab-panel.p1 .star-card { background: var(--p1-bg); border-left-color: var(--p1-accent-soft); }
.tab-panel.p1 pre { border-left: 4px solid var(--p1-accent-soft); }
.tab-panel.p1 .skill-card.star { ... }  /* P2 同款换变量 */
```

**覆盖范围必须够全**:h2/h3/h3::before/h4/philosophy/em/strong/表头/star-card/skill-card/pre 左边框/combo 标题 —— 漏一个切 Tab 就视觉割裂。

**Tab 必须有明显的「可点击切换」提示**(否则用户看不出能切):标签上方一条虚线提示条说明「可点击切换的双标签」;每个 `.tab-btn` 顶部用 `::before` 放状态小字 —— 未选中 = `▸ 点击切换到这一场`、选中 = `● 当前查看`(随 `.active` class 自动切,不能写死在 HTML 里);未选中标签 hover 上浮 + 投影,选中标签 `cursor: default`。

## 四、资源清单结构(陆节,协作核心区)

### 6 大资源类型

一、动作(Animation/Spine) 二、特效(VFX) 三、音效(SFX/Voice/BGM) 四、UI/HUD 五、场景/演出 六、过场 Timeline/Prefab

每类下:`X.1 双场全复用` + `X.2/X.3 各场独家新增`。顶部一排 `.res-summary` 汇总卡片,大数字 = 各类新增总数(动态统计)。

### 表格列结构(JS runtime 注入后的最终形态)

```
# | 动作 | 资产名称 | ...该表原有数据列... | 📚 参考 | 📎 产出/协作 | 操作
```

| 列 | 来源 | 谁填 | 编辑方式 |
|---|---|---|---|
| #、动作、原数据列 | HTML 静态写 | 设计师 | 普通数据列,点「编辑」按钮才可改 |
| **资产名称** | JS 注入(动作列后 index 2) | **策划/配置者** | 普通数据列,点「编辑」按钮才可改;视觉同普通列(不加特殊底色) |
| **📚 参考** | JS 注入(倒数第三) | **策划/配置者** | **始终可编辑**(蓝调底色),给美术的参考材料 |
| **📎 产出/协作** | JS 注入(倒数第二) | **美术/特效/音效/UI** | **始终可编辑**(金调底色),贴产出 |
| 操作 | JS 注入(最后) | — | 编辑/划除按钮 |

**关键区分**:始终可编辑的列(参考/产出)有特殊底色 + 占位提示;只能点「编辑」改的列(动作/资产名称/数据列)视觉同普通列,编辑态才统一加浅黄底 + 虚线框。

### 静态表格只写「原有数据列」,其余 JS 注入

HTML 里每个 `<table class="res-table">` 只写 `# | 动作 | [原数据列] | 用途/说明`。资产名称/参考/产出/操作 四列全部 JS runtime 注入。**不要在表头硬写「类型」「距离」这种能从标题/说明推断的列**(标题已写「全新增」就别再来一列类型)。

### 复用表特殊规则

紧邻 h4 含「复用」二字的表(如 1.1 双场全复用):走 [组件实现 § 二「复用资源视觉降权」](review-html-components.md) 自动识别 → 整表加 `.reuse-table` 类,获得沙青色调 + contrib/ref 列 opacity 0.5 + 占位文字 override 成「— 复用 · 无需新产出」。**操作列正常注入(可编辑/划除),「+新增资源」按钮也正常注入** —— 评审过程会发现「还有别的复用资源该归这里」,必须能直接加;新增的复用行天然继承 `.reuse-table` 的视觉降权与占位语义,无需特殊处理。

> 早期版本曾经做过「复用表不注入新增按钮」的硬约束,实战发现「不让加」过于教条,移除该约束。

## 六、协作工作流(导出 → 回传 → 加载)

`.collab-actions` 协作工具栏收纳在**左侧坞** `.left-dock` 内(详见 [组件实现 § 二「左侧坞」](review-html-components.md)),宽屏垂直居中稍偏下,窄屏 fallback 到右下角:

- **📤 导出修改版** → 收集 sessionStorage(含 `__history__` / `__editor__` 全套) + IndexedDB,嵌入 `<script id="__embed_state__">`(写 `window.__EMBED_STATE__` / `__EMBED_IDB__` / `__EMBED_META__`),基于 `window.__ORIGINAL_HTML__`(init 前的原始 DOM 快照)重建 HTML,触发下载。
  - 文件名带日期 + 导出人名;导出 prompt 的「姓名」默认填 `__editor__` 当前值(`window.__getEditorName()`),不用每次重打
  - **导出 = 历史一起打包带走**,收件人打开即可看到完整变更轨迹
- **♻️ 重置本地** → 清空 sessionStorage + IDB + reload(美术拿到新版时清旧痕迹)

**闭环**:
```
设计师 → 改完点导出 → 文档_日期_设计师.html → 发美术
美术 → 打开(自动加载内嵌数据)→ 在「📎 产出」列贴产出 → 导出 → _美术.html → 回传
设计师 → 打开美术回传的 → 自动加载 → 看到产出 → 继续迭代
```

`window.__ORIGINAL_HTML__` 必须在所有 init 之前备份(`'<!DOCTYPE html>\n' + document.documentElement.outerHTML`),保证导出的是干净 baseline + 内嵌数据,不含 runtime 注入的 DOM。

⚠️ **从零重写交互层的人必读**:协作闭环有 4 个致命陷阱(IDB / embed 脚本 / LS 恢复时序 / TR 漂位),详见 [组件实现 § 五](review-html-components.md)。

## 七、反模式自检

❌ markdown/docx 输出设计稿 — 承载不了交互,审稿不便
❌ 用 localStorage 而非 sessionStorage — 「不导出也保存」违背协作语义,易误操作
❌ 忘备份 `__ORIGINAL_HTML__` — 导出的 HTML 含 runtime 注入 DOM,再打开重复注入
❌ 表头硬写「类型/距离」等可推断列 — 标题已写「全新增」就别再来一列
❌ 资源清单不分类型 / 不分复用-新增 — 美术对资源找不到自己那部分,工作量估算不清
❌ 始终可编辑列 与 点编辑才可改列 视觉不分 — 用户分不清哪些能直接填
❌ 数字硬编码不动态 — 划除资源后数字不变,失去审稿意义
❌ 双场 Tab 调色覆盖不全 — 切场视觉割裂
❌ 媒体不包 `.media-wrap` / 不设 `draggable=false` — 浏览器原生拖拽把视频当 HTML 复制进格子,撑爆存储
❌ pre 文本流堆技能/AI/连招 — 信息密度高扫读慢,要卡片化/流程图/距离条
❌ 大视频内嵌 base64 — 文件膨胀到几百 MB,优先贴链接(网盘/B站/YouTube)
❌ CSS 变量散落硬编码 — 改调子要全文搜替换
❌ 资源 tier 只有 1-2 项就用 `<ul>` 而非 `.res-table` — `updateAllRowCounts` 只数 `.res-table` 行,`<ul>` 项会被汇总卡片 / 章节数字静默漏算;少量项也用 `.res-table`
❌ 复用资源段视觉跟新增段一样 — 美术认领时把复用项误以为要做的产出,工作量估算错乱;**JS 自动识别 h4 含「复用」且不含「新增」** → 沙青色调降权(详见 组件实现 § 二)
❌ 浮动目录只有一级 — 文档长了找子标题靠滚屏;**两级树**自动扫每个 section 的 h3 作为子项,chevron 收起展开(详见 组件实现 § 二)
❌ 浮动目录、协作工具栏、置顶按钮各自固定到不同角落 — 用户视觉跨度大,鼠标到处跑;**全部塞进 `.left-dock` 垂直居中** 一处包圆,扫一眼就知道哪里能点
❌ 章节导航子项落在非激活 Tab 时硬跳 `#h3-id` — 浏览器跳到隐藏 panel 的 h3 看不到内容(panel `display: none`);**子项点击拦截**,先手动切 tab class 再 `setTimeout 50ms` scrollIntoView
❌ 文案里混用英文半角标点 `,` / `.` / `:` / `;` / `(` / `)` / `"` —— 在中文段落里显得拥挤、夹生,破坏古风排版节奏;**所有文案标点统一中文全角**(`，。：；（）""`);例外只有文件名 / 路径 / URL / 代码 / 数值+单位 / 英文专有名(详见 § 二·b 文案标点规范)

**JS / 协作层的代码级陷阱(IDB 收口 / embed 脚本拼接 / LS 时序 / TR 漂位 / 改动历史基线对比 / 三层去重)**:详见 [组件实现 § 五](review-html-components.md)。
