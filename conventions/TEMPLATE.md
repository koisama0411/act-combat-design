# 项目战斗约定模板 · Project Combat Conventions Template

> **这是什么 / What is this**
> 一份**留空的填写模板**。任何 ACT 战斗设计开始前，先用它把"你这个项目"的战斗约定定下来——招式集、分类轴、视觉预警、受击状态、帧率、命名、配队、养成……
> 本方法论**不预设**任何一套战斗体系；这些都由你的项目自己决定。下面每个维度给出**业界常见的设计空间（菜单）**供你挑选 / 改造，再把你的选择填进「📝 你的项目」一栏。
>
> **怎么用 / How to use**
> 1. 把本文件复制一份到你的项目里（如 `your-project-conventions.md`）。
> 2. 逐维度读"设计空间"，在「📝 你的项目」填你的选择（删掉不适用的菜单项也行）。
> 3. 设计角色 / Boss 时，把填好的这份**和本 skill 一起喂给 AI**（或让 AI 帮你填）。AI 会严格按你的约定来，而不是套用别的项目。
> 4. 想看一份**填好的范例** → [example-conventions.md](example-conventions.md)（配套示例角色「苍岚」用的虚构项目）。
>
> 不是每一项都必填。单机买断角色动作游戏可跳过"配队 / 抽卡养成"；纯单角色游戏可跳过"分类轴"。按需取用。

## 1. 项目基调与视角 · Genre & Perspective

**为什么先定它**：视角和帧率直接决定几何（AOE 怎么摆、走位有几个维度）和动作帧数预算，是后面一切的地基。

**设计空间**：
- **视角**：2D 横版 / 2.5D / 3D 第三人称 / 俯视角(ARPG) / 固定镜头
- **维度**：横版只有 X 轴（+ 极少纵深 Z），俯视 / 3D 有完整平面走位——AOE 布局思路完全不同
- **屏幕 / 场景尺度**：横版常用"屏宽 = N 格 / N 米"；3D 用世界单位。先定一个基准距离单位
- **帧率基准**：30fps（很多二游 / 手游）/ 60fps（主机动作）—— 所有"前摇 X 帧"都相对这个基准

**📝 你的项目**：
- 视角 / 维度：________
- 距离单位 / 屏幕尺度：________
- 帧率基准：________ fps

> 关联：2D 几何与 AOE 布局思路、3D Boss 走位差异 → 见 [boss-ai-and-arena.md](../references/boss-ai-and-arena.md)

## 2. 招式集 · Move / Skill Taxonomy（最重要）

**为什么**：这决定"每个角色至少要有哪些招"。它是你项目的战斗"骨架契约"，美术 / 程序按它估工作量。**不同项目差别极大**——硬核动作游戏可能十几类招，轻量手游可能只要 4-5 类。

**设计空间（ACT 常见招式类型菜单，按需选用，不是必须全有）**：

| 招式类型 | 说明 | 常见于 |
|---|---|---|
| 普攻连段 | 轻攻击 N 连，起手轻 / 末段重 | 几乎所有 ACT |
| 重攻击 / 蓄力攻击 | 长按蓄力，单独按键或长按普攻 | 黑神话、只狼、剑星 |
| 核心特殊技 | 角色机制载体（蓄力 / 变形 / 资源消耗…） | 几乎所有 |
| 闪避 | 翻滚 / 步伐，可带完美闪避窗 | 几乎所有 |
| 闪避后续（闪避攻击 / 反击） | 闪避后接的攻击 / 极限闪避反击 | 鸣潮、绝区零 |
| 防御 / 格挡 / 弹反 | 招架、拼刀、识破 | 只狼、卧龙、战神 |
| 空中攻击 | 跳跃 / 滞空攻击 | DMC、猎天使 |
| 终结技 / 大招 | 能量满后的高爆发演出 | 几乎所有 |
| 团队联动技 / 切人技 | 多角色协同时的支援 / 接管招（仅团队制需要） | 原神、崩铁、ZZZ |
| 处决 / 抓投 | 特定条件触发的强力单段 | 忍龙、战神 |

**要决定的**：
- 你项目**每个角色必备**哪几类？（= 设计每个角色时逐类要填的清单）
- 哪些是**可选 / 角色特色**？
- 招式之间的**衔接 / 取消规则**（连段能否被闪避取消？蓄力能否被打断？）

- ________
- ________
- ________

## 3. 角色分类轴 · Character Classification Axes

**为什么**：给角色贴标签，让团队一眼读懂定位、让玩家快速理解。**单角色游戏可跳过。**

**设计空间（常见分类轴，按需组合）**：
- **战斗职业 / 风格**：如 力量型 / 技巧型 / 防御型；或按武器分（剑 / 枪 / 拳 / 法）
- **队伍定位**：输出主C / 副C / 坦克 / 治疗 / 辅助增益 / 控制击破
- **稀有度 / 强度档**（抽卡游戏常用）：如 A / S / SSR；或 3★/4★/5★
- **属性 / 元素**（如有元素系统）：火 / 水 / 雷 / …
- **上手难度**：★1-3

**📝 你的项目 · 分类轴**：
- 轴 1：________（取值：________）
- 轴 2：________（取值：________）
- 轴 3（可选）：________

> 关联：分类如何影响机制取舍（稀有度 = 复杂度上限等） → 见 [character-archetypes.md](../references/character-archetypes.md)

## 4. 视觉预警契约 · Telegraph Convention

**为什么**：玩家靠"看见预警 → 反应"来博弈。你必须定一套**全项目统一**的视觉语言：什么颜色 / 特效 = 必闪？什么 = 可格挡 / 可弹反？这是 Boss → 玩家的契约，乱了玩家就背板。

**设计空间（业界做法）**：
- **二元**：一种"危险预警"（必须应对） vs 普通攻击（只狼的"危"字）
- **三元 / 多色**：不同颜色对应不同应对方式（如 红=必闪、黄=可格挡、蓝=可弹反 / 识破）
- **形状 / 音效辅助**：圈 / 线 / 扇形指示范围 + 专属警示音
- **覆盖时长规则**：预警必须覆盖整个前摇，且足够长给反应窗

**要决定的**：用几种预警？每种对应什么应对（闪 / 格 / 弹 / 打断）？预警的视觉表现是什么？

**📝 你的项目 · 预警契约**：
- ________ = ________（应对：________）
- ________ = ________（应对：________）

> 关联：telegraph 帧数哲学、颜色语义化 → 见 [meta-rules.md](../references/meta-rules.md) 与 [anti-patterns.md](../references/anti-patterns.md)（"telegraph 模糊必背板"反模式）

## 5. 受击 / CC 状态分类 · Hit-React & Crowd-Control Taxonomy

**为什么**：所有"打出去的控制效果"要成体系，否则衔接混乱（A 招把敌人击飞、B 招却假设敌人在地面）。需要一套**有限的、命名清晰的**受击状态 + 它们之间的衔接规则。

**设计空间（常见受击状态）**：
- 硬直 / 踉跄（极短，原地）
- 击退（水平后退，不离地）
- 浮空（短暂离地，可接浮空连段）
- 击飞（大幅离地飞远）
- 倒地 / 击倒（需起身的状态）
- 眩晕 / 冻结 / 击破（架势 / 韧性条耗尽的特殊可乘之机）

**要决定的**：用哪几种？各自持续多久？**衔接规则**（已浮空目标再被某招会怎样？击退能否衔接浮空？）

**📝 你的项目 · 受击状态**：
- ________（持续：____，衔接：____）
- ________（持续：____，衔接：____）

> 关联：击破 / 架势 / 韧性系统通用公式 → 见 [numbers-and-progression.md](../references/numbers-and-progression.md)

## 6. 命名规范 · Asset Naming Convention

**为什么**：动作 / 特效 / 技能资产的命名要统一，美术 / 程序 / 策划才能对齐。

**设计空间（命名只需做到「统一 + 一看就懂是谁的哪招哪段」，风格自选；下面给两种常见风格对照，别把任一种当默认）**：
- 风格 A（snake_case）：动作 `attack_01` / `skill_charge_pre·loop·release·recover` / `dodge` / `parry_start·success·attack` / `ultimate_01`；特效 `vfx_<角色>_<招式>_<段位>`
- 风格 B（PascalCase，本仓示例角色苍岚用这套）：动作 `Attack_01` / `Skill_Charge_Pre·Loop·Release·Recover` / `Dodge` / `Parry_Start·Success·Attack` / `Ultimate_01`；特效 `Vfx_<角色>_<招式>_<段位>`
- 子段后缀（通用）：`_pre` / `_charge` / `_loop` / `_release` / `_recover` / `_air` / `_start` / `_end`

## 7. 团队 / 配队模型 · Team & Synergy Model（团队制才需要）

**为什么**：如果是多角色组队玩法，需要定"队伍怎么组、角色怎么协同、有没有联动 / 切人机制"。**单角色游戏跳过。**

**设计空间**：
- **队伍规模**：单角色 / 双人 / 三人 / 更多
- **协同机制**：纯属性叠加 / 元素反应 / 切人联动技 / 共享资源槽 / 羁绊系统 / 后台挂状态
- **主轴**：靠主C一个人打 / 多核循环 / 队伍机制结算
- **角色协作模式**：是否有"角色离场后的延续行为"（如召唤体、驻场输出、切人接管）

**📝 你的项目 · 配队模型**：

- 协同机制：________
- 主轴 / 出伤思路：________

## 8. 养成 / 成长模型 · Progression Model

**为什么**：养成方式（尤其抽卡命座）会反过来影响角色机制设计（要不要留"命座扩展机制"）。

**设计空间**：
- **买断单机**：技能树 / 装备 / 符文，无付费墙 → 机制一次给全
- **抽卡 / gacha**：等级 + 命座 / 星魂 / 影画等"叠影"系统，需设计 C0 完整体验 + 命座扩展机制 + 防"必抽满命"
- **赛季 / Roguelite**：每局临时 build，强调流派组合

## 10. 可选状态 / 元素轴 · Optional Status / Element System

**为什么**：有些项目用一套"状态 / 元素工具箱"（如元素反应、异常状态、属性相克）给角色加辨识度。**非必须**，前期可少用，需要机制辨识度时再引入。

**设计空间**：元素反应（原神）/ 异常积累（绝区零紊乱）/ 属性相克 / 自定义状态轴（内伤·重压·破甲…）
